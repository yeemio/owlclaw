# Console 浏览器测试要求（测试总监签发）

> **签发**：测试总监  
> **适用范围**：OwlClaw Web Console 发布前必过质量门  
> **版本**：v1.0  
> **日期**：2026-03-04

---

## 一、强制要求（Mandatory）

### 1.1 发布前必须满足

Console 合并到 main 或对外发布前，**必须**通过以下门禁，否则不得放行：

| 门禁 | 要求 | 验收方式 |
|------|------|----------|
| **无 500 白屏** | 任意页面在无 DB / 无 Hatchet / 无 Langfuse 场景下，不得出现未捕获 500 或白屏 | 人工或自动化逐页访问，确认有内容或友好空状态 |
| **API 契约一致** | 前端调用的 API 路径、参数、响应解析与 Backend 契约一致 | 对比 OpenAPI / 实际请求与 `console-backend-api` 设计 |
| **关键路径可走通** | Overview → Governance → Ledger → Agents 四条主路径，用户可完成"打开 → 查看 → 筛选/分页" | 真实用户或自动化执行，记录通过/失败 |
| **无敏感信息泄露** | Network / Console 无 token、密钥、内部路径泄露 | DevTools 检查 |

### 1.2 禁止行为

- **禁止**仅凭"点开菜单能打开"即判定通过
- **禁止**跳过无 DB / 无外部服务场景的验证
- **禁止**在无证据（截图/请求日志/报告）的情况下签字放行

## 二、测试维度与最低覆盖（Minimum Coverage）

测试总监要求的四个维度，**每个维度至少覆盖指定项**，缺一不可：

| 维度 | 定义 | 最低覆盖 | 证据要求 |
|------|------|----------|----------|
| **功能** | 业务能力是否按 spec 实现，用户能否完成预期操作 | 每页≥1 个核心操作（筛选/切换/展开/刷新）被验证 | 操作步骤 + 预期 vs 实际 |
| **API** | 契约、参数、错误码、降级策略 | 每端点≥1 次成功 + 1 次边界/错误（如 404、422、无 DB） | 请求/响应摘要或 curl 输出 |
| **代码/网络** | 请求链正确、无多余/错误请求、无 JS 异常 | 每页加载时 Network 无 4xx/5xx（除预期 404），Console 无未捕获错误 | Network 截图或 HAR、Console 日志 |
| **效果** | 视觉、空状态、性能、可访问性 | 暗色主题一致、空状态友好、首屏可交互时间可接受 | 截图或简要描述 |

### 2.1 维度不足时的处置

- 某维度缺项 → 测试不通过，补齐后方可放行
- 某维度全部通过但发现 P1 缺陷 → 可附带修复计划放行，但缺陷须录入并跟踪

## 三、测试类型与触发时机

| 类型 | 触发时机 | 执行方 | 产出 |
|------|----------|--------|------|
| **冒烟（Smoke）** | 每次 console 相关 PR 合并后、每日构建 | CI 或值班 | 通过/失败，≤15 分钟 |
| **回归（Regression）** | 发版前、每月 | 人工或 Playwright | 完整报告，按维度记录 |
| **探索（Exploratory）** | 新页面/新 API 上线后、重大重构后 | 人工 | 缺陷列表 + 改进建议 |

### 3.1 冒烟必须覆盖

- 服务可启动，`/console/` 返回 200
- 4 个主页面（Overview、Governance、Ledger、Agents）可打开且无白屏
- `/api/v1/agents` 无 DB 时返回 200 且含 `message`

## 四、可追溯性（Traceability）

| 要求 | 说明 |
|------|------|
| **Spec 对应** | 每个测试项须可追溯到 `console-frontend/requirements.md` 或 `console-backend-api/design.md` 的条目 |
| **缺陷闭环** | 测试发现的缺陷须有 ID，修复后须在报告中标明「已修复」并复验 |
| **报告留存** | 每次回归报告须存入 `.kiro/reviews/`，命名含日期，如 `YYYY-MM-DD-console-browser-verification.md` |

---

## 五、交付物（Deliverables）

测试执行方须交付：

1. **执行报告**：按维度列出每项通过/失败，含证据索引（截图、HAR、curl 命令）
2. **缺陷列表**：未修复缺陷的 ID、严重程度、复现步骤
3. **放行建议**：通过 / 有条件通过（附条件）/ 不通过

报告模板见 `docs/console/BROWSER_VERIFICATION_CHECKLIST.md` 末尾。

## 六、风险分级与优先序

| 级别 | 定义 | 示例 | 处置 |
|------|------|------|------|
| **P0** | 阻断发布，用户无法使用 | 500 白屏、API 全挂、敏感信息泄露 | 必须修复，不得带缺陷发布 |
| **P1** | 严重，影响核心场景 | 主流程筛选/分页无效、空状态显示错误 | 发版前修复或明确修复计划 |
| **P2** | 一般，有替代方案 | 某子功能异常、非关键页展示问题 | 纳入 backlog |
